## Logging — conclusions from 2026-03-28 discussion

### Intended design (current repo)

- **`extdeps/run-with-stdout-logged.py`**: spawns **`python app.py`**, reads its stdout/stderr pipe, appends to **`LOG_PATH`** (e.g. host `.../onboard.log`), writes `__run-with-stdout-logged__:` header lines on open.
- **`run.sh`**: starts **`twoway.py`** (and `ui_server.py`) in the **background**, then **`exec`**’s the wrapper around **`app.py`** only.
- **`common.log`**: **`StreamHandler(sys.stdout)`** in each process — no direct `FileHandler` on `LOG_PATH`.

So **by design**, **`onboard.log` should be mostly `app.py` lines**; **`twoway` should show up in `docker logs thermo-onboard`**, not necessarily in the bind-mounted file.

### Observed on Pi (`pizero`)

- **`/var/log/thermo-onboard/onboard.log`** contained **both** `twoway:` and `app:` lines (interleaved).
- **`head -5`** showed **no** `__run-with-stdout-logged__:` headers — may mean **older wrapper**, **rotation**, or **not** the file the current wrapper creates first.

### Interpretation

- **Not “overwrite”**: two writers **append** to the same path → **interleaving**.
- For **`twoway:`** to appear **on disk** in that file, **twoway’s stdout** must ultimately hit **that** append path (or an equivalent). Stock **`CMD ["bash","./run.sh"]`** does not redirect bash stdout to the log file; **if** the Pi still shows mixed content, **verify deployment** (image digest, custom CMD, redirects, or wrappers).

## Reference commands (quick)

- **Bind-mounted app log**: `tail -f /var/log/thermo-onboard/onboard.log` (or `THERMO_LOG_DIR` override).
- **Docker stream** (includes background processes): `docker logs -f thermo-onboard`.
- **Internal state**: `curl -sS http://127.0.0.1:8080/manage` (adjust if needed).
